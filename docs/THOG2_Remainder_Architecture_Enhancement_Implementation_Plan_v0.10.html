1. Executive summary

The implementation proceeds in six stages. Stages 0–2 establish correctness and whether the remainder can learn. Stage 3 develops the memory-clean custom optimiser. Stage 4 determines whether the architecture deserves completion. Stage 5 is entered only after a positive scientific result.

Critical path: exact decoding → exact materialised-weight addressing → exact gradient decomposition → reference AdamW proof → memory-clean custom optimiser.
StagePrimary outcomeExit decision
0Contracts, decoder, projector and offline diagnosticCore maths and indexing proven
1Forward-only int8 remainder pathZero equivalence and address correctness
2One-backprop routing and R1 reference optimiserRemainder can demonstrably learn
3R2 factored-RMS stochastic optimiserLean optimiser works and resumes exactly
4Matched controlled sweepProceed, revise optimiser, or stop
5Packed storage and optional streamed autogradProduction memory path
Central caution. The custom optimiser is the largest implementation unknown. It shall not be developed in final packed form until the deliberately expensive reference optimiser proves the representation useful.

2. Implementation principles

  1. Preserve the existing path. remainder_mode=off bypasses all new storage, hooks and optimiser code.
  2. Test before code. Commit a stage-specific test plan before implementing that stage.
  3. Separate science from optimisation engineering. R1 may be wasteful; R2 may not.
  4. Use one canonical matrix layout. Remainders are row-major over final (out_features, in_features) matrices.
  5. Do not double-apply coefficient gradients. Hook mode returns the projected weight gradient to existing autograd.
  6. Fail loudly. Unsupported modes, geometries and checkpoint combinations raise explicit errors.
  7. Measure peak and persistent memory separately.

2.1 Code and branch policy

Existing nanoGPT lines shall not be deleted; superseded lines shall be commented out. New THOG2 code shall use the established THOG markers. All work remains on remainder_architecture_enhancement unless explicitly directed otherwise.

4. Stage 0 — Contracts and test harness

Establish the contracts every later stage depends on.

Pre-code deliverable

docs/THOG2_Remainder_Architecture_Enhancement_Stage_0_Test_Plan_v0.10.html.

Implementation

  1. Add configuration and validation with default mode off.
  2. Define the six legal tensor-family keys and canonical final-matrix views.
  3. Implement literal repeated-nextafter decoding for FP16, BF16 and FP32.
  4. Implement directional code-step calculation and bound masking.
  5. Add project_weight_gradient().
  6. Implement CHEBY projection adapters for every existing spectral geometry using the exact forward basis matrices and axis order.
  7. Build the fixed-width offline diagnostic for 1, 2, 3, 4, 6 and 8 bits.

Tests and exit

  • Bitwise decoder tests including zero, subnormals, exponent boundaries and max finite values.
  • Address stability across qkv/head reshaping.
  • Projection reconstruction and orthogonality tolerances from v0.12.
  • Every registered geometry covered.
  • All Stage 0 and existing tests green; diagnostic deterministic and machine-readable.
Blocker. Projection must be the exact reverse contraction of the active materialiser, not a shape-based approximation.

5. Stage 1 — Forward-only remainder path

Add int8 storage and effective-weight construction without learning.

Pre-code deliverable

docs/THOG2_Remainder_Architecture_Enhancement_Stage_1_Test_Plan_v0.10.html.

Implementation

  1. Add zero-initialised int8 remainder buffers for covered spectral matrices.
  2. Add shape, layout and tensor-key metadata.
  3. Implement the straight-through effective-weight builder.
  4. Integrate the six required matrix families.
  5. Add checkpoint save/load.

Tests and exit

  • Active mode with zero codes reproduces the base path.
  • Single-position mutation changes exactly one intended matrix element.
  • All families and first/middle/final layers covered.
  • Checkpoint round-trip exact.
  • CPU, scruffy BF16 and dreedle FP16 smoke tests pass.

6. Stage 2 — Gradient routing and R1 reference optimiser

Establish the one-backprop architecture and determine whether the remainder can learn with familiar, deliberately expensive optimisation.

Pre-code deliverable

docs/THOG2_Remainder_Architecture_Enhancement_Stage_2_Test_Plan_v0.10.html.

Implementation

  1. Attach effective-weight gradient hooks.
  2. Compute G_projected and G_remainder.
  3. Return only G_projected to autograd.
  4. Verify coefficient gradients against direct analysis transforms.
  5. Add FP32 remainder masters and AdamW reference groups.
  6. Convert rejected gradients to code-space gradients through directional code steps.
  7. Quantise masters to bounded int8 values after each update.
  8. Add R1 checkpoint, resume and telemetry.
  9. Require gradient accumulation equal to one.

Tests and exit

  • Gradient decomposition reconstructs the effective gradient.
  • Hook gradient and explicit projection produce the same coefficient gradient.
  • No gradient is applied twice.
  • R1 descends deterministic toy objectives.
  • Fresh → save → resume matches uninterrupted execution.
  • Atomic non-finite skip leaves both paths unchanged.

Mini-gate

Run matched widths 0, 2 and 4. Stage 3 proceeds only if width 4 gives credible improvement and rejected gradients are non-trivial.

Interpretation. If R1 fails, stop. The representation or routing is at fault; the future custom optimiser cannot rescue it.

7. Stage 3 — R2 target optimiser

Implement the factored-RMS stochastic integer optimiser. This is the largest workstream and is not a small AdamW subclass.

Pre-code deliverable

docs/THOG2_Remainder_Architecture_Enhancement_Stage_3_Test_Plan_v0.10.html.

MilestoneWorkProof
3ADirectional step, bound masking and FP32 code gradientFinite-difference and sign tests
3BRow/column second moments and RMS clippingIndependent dense-reference match
3CSeeded stochastic integer updateExpected movement and reproducibility
3DAccumulation, skip, checkpoint and coordinationResume equivalence and CUDA smoke

Implementation

  1. Store only row and column FP32 second-moment vectors per matrix.
  2. Implement exact v0.12 equations and defaults.
  3. Use a dedicated seeded device RNG and checkpoint its state.
  4. Implement transient pending integer steps across accumulation microsteps.
  5. Average factor observations across the accumulation window.
  6. Apply values and factor state only after a complete finite update.
  7. Restore pre-update RNG on skipped updates.
  8. Report actual and theoretical packed persistent bytes.
  9. Add update frequency, saturation and factor telemetry.

Tests and exit

  • Factored normaliser matches an independent reference.
  • Mean stochastic update matches requested movement.
  • Seeded runs and resumes are reproducible.
  • Accumulation has the expected distribution.
  • Outward bound updates are impossible.
  • Skipped updates restore values, factors and RNG.
  • R2 trains stably on toy and short OWT runs.

8. Stage 4 — Controlled scientific evaluation

Determine whether the architecture merits completion. This is an experiment package, not optimiser development.

Pre-run deliverable

docs/THOG2_Remainder_Architecture_Enhancement_Stage_4_Test_Plan_v0.10.html, including exact run matrix, seeds, token budget and analysis.

ComparisonWidthsModesSeeds

Optimiser gap2, 4R1 vs R23

Controls and outputs

  • Identical geometry, basis, orders, initial base coefficients, data order, token budget and base AdamW schedule.
  • Validation loss versus bit width and R1/R2 gap.
  • Saturation, non-zero fraction, projected/rejected norms and update norms.
  • Persistent memory, peak memory, tokens/s and update time.

Decision

  • Proceed: 4 bits passes and R2 is credible.
  • Optimiser research: R1 passes materially but R2 does not.
  • Stop: R1 fails at 4 bits.

9. Stage 5 — Packed storage and streamed backward

Enter only after Stage 4 passes.

Pre-code deliverable

docs/THOG2_Remainder_Architecture_Enhancement_Stage_5_Test_Plan_v0.10.html.

Packed storage

  1. Pack fixed-width signed values into torch.uint8.
  2. Implement sign extension and exact unpacking for 1, 2, 3, 4, 6 and 8 bits.
  3. Update and repack blockwise without a persistent unpacked shadow.
  4. Implement an accelerated decoder bitwise-equivalent to the reference decoder.
  5. Add packed checkpoint schema and int8-to-packed conversion.

Streamed autograd decision

Measure the full-matrix-hook backend first. Implement streamed custom autograd only if peak memory violates the approved threshold or blocks intended model sizes.

Streamed backend

  1. Tiled effective-gradient recomputation.
  2. Pass A accumulates coefficient gradients and factor observations.
  3. Pass B rematerialises projected tiles, derives rejected tiles and generates pending integer updates.
  4. Publish coefficient gradients once and discard each tile immediately.

Exit

  • Packed and int8 logical values are bitwise equivalent.
  • Hook and streamed gradients match; stochastic updates match statistically.
  • Checkpoint conversion and resume pass.
  • Persistent optimiser overhead meets v0.12.
  • Measured peak-memory gain justifies complexity.

10. Cross-cutting requirements

Configuration and lifecycle

  • Add fields to canonical config and parser; expose wrapper controls only after implementation.
  • Preserve wrapper defaults with remainder off.
  • Include structural and optimiser settings in compatibility signatures and manifests.
  • Fresh, resume and fork follow the existing enhanced lifecycle.
  • RNG, values and optimiser state participate in exact resume.
  • Checkpoints occur only at completed update boundaries.

Performance discipline

  • No Python loop per materialised weight in training.
  • Literal repeated-nextafter is reference-only.
  • Profile vectorised code before custom kernels.
  • Any CUDA/Triton path requires an independent CPU reference.
