thog2 · Remainder Architecture Enhancement1 · Executive summary
1.0 Executive summary

Two learned objects.
One effective weight field.

THOG2 presently learns compact spectral coefficients and recreates transformer weights from them. The proposed enhancement adds a second learned object containing signed, low-order precision offsets at materialised weight positions.

The transformer still consumes exactly one value at every weight position: effective weight = spectrally generated base weight + remainder offset.
Base object

Spectral coefficients

Compact, structured and basis-expressible. Existing THOG2 materialisation and AdamW path remain intact.

Remainder object

Low-bit signed values

Materialised-weight-indexed local adjustments intended to supply precision the base geometry cannot independently express.

Central hypothesisTHOG2 may already be numerically close to useful dense weights and may be missing only a small number of independent low-order bits at each materialised weight position.
Central riskThe remainder may require too many bits, or its optimiser may require persistent per-weight state large enough to erase the compression saving.

The architecture is dual-object in storage and optimisation, but single-object in execution.

thog2 · Remainder Architecture Enhancement2 · Concept
2.0 Concept

A local precision affordance over a structured base

The remainder path is not a residual connection, not a second dense model and not a replacement basis. It is a separate low-bit learned object whose sole function is to make local corrections to materialised base weights.

base_weight = materialise(base_coefficients) remainder_offset = decode(remainder_value, base_weight, effective_dtype) effective_weight = base_weight + remainder_offset
Base role

Structure

Captures content expressible through the selected spectral geometry and orders.

Remainder role

Local precision

Supplies signed low-order adjustments at individual materialised weight positions.

Transformer view

One weight

Attention and MLP operations consume only the summed effective weight.

2.1 Basis independence

The remainder path is associated with materialised weight positions rather than with Chebyshev coefficients. It must therefore remain usable with Chebyshev, DCT and future registered bases, provided they materialise the same canonical weight field.

2.2 Scope boundary

Sheet-geometry enhancement is outside this document. The present specification does not invalidate that work, but the remainder architecture is treated as the more direct response to the observed shortage of independent local resolution.

Controlled descriptionThe remainder object stores signed low-order precision offsets, indexed by materialised weight position.

thog2 · Remainder Architecture Enhancement3 · Object association
3.0 Dual learned objects

The weight position is ground truth

The two learned objects are not paired by coefficient index. They are associated through the canonical materialised weight position.

materialised_weight_position ├── base_weight generated from spectral coefficients ├── remainder_value loaded from the remainder object └── effective_weight base_weight + decoded remainder_offset

3.1 Canonical addressing

Ordinary matrix form(layer, tensor_family, row, column)
Head-aware form(layer, tensor_family, head, role, row, column)

3.2 Required invariants

  1. Every generated materialised weight position shall have exactly one base value.
  2. Every remainder value shall map to exactly one materialised weight position.
  3. A position without an active remainder shall have an explicit zero remainder.
  4. Vectorisation, fusion, qkv packing, head splitting and materialisation order shall not change the logical association.
  5. Checkpoint reload shall restore the association exactly.
Lethal failure modeA remainder value applied to the wrong layer, tensor family, head, row or column creates plausible-looking but meaningless training. Address correctness requires dedicated single-position mutation tests.

4.1 Forward pass

base_weight = basis.materialise(base_coefficients) remainder_value = remainder_store[position] remainder_offset = decode(remainder_value, base_weight, effective_dtype) effective_weight = base_weight + remainder_offset output = transformer_operation(input, effective_weight)

4.2 Loss

The loss is unchanged. There is no separate remainder objective in the baseline architecture.

loss = language_model_loss(outputs, targets)

4.3 Backpropagation

There is one backward pass. Autograd obtains the gradient of loss with respect to the effective weight while also propagating gradients to earlier activations.

G_effective = ∂loss / ∂effective_weight
Major architectural simplificationThere is no second model, second forward pass, second loss or second backpropagation. The two learned objects diverge only after the single effective-weight gradient has been obtained.

thog2 · Remainder Architecture Enhancement5 · Gradient treatment
5.0 Gradient treatment

Chebyshev gets first bite.
The remainder gets the rejection.

The base and remainder must not both chase the entire effective-weight gradient. The baseline routing rule is orthogonal projection onto the base geometry followed by subtraction.

G_base = P_base(G_effective) G_remainder = G_effective - G_base

For an orthonormal basis matrix B:

G_base = B (Bᵀ G_effective) G_remainder = G_effective - B (Bᵀ G_effective)
First bite

Base path

The component expressible by the selected basis is converted into coefficient gradients and handled by the existing THOG2 AdamW path.

Crumbs

Remainder path

The rejected component supplies the materialised-weight-indexed signal used by the remainder updater.

Important qualificationThis is a gradient-level division of labour. AdamW later rescales the base update using moment history, so perfect non-interference is not guaranteed. Update-level reconciliation is a later option, not a v0.10 requirement.

thog2 · Remainder Architecture Enhancement6 · Numeric semantics
6.0 Numeric meaning

Signed low-order precision offset

The remainder value is a signed low-bit integer. Its numeric meaning must be mechanically tied to the least significant representable place of the selected effective-weight dtype at the local exponent.

Baseline definitionOne remainder unit equals one ULP-like least-significant-place step of the effective-weight dtype at the base weight’s local scale. It is not a learned scale and it does not default to 1.0.
precision_step(base_weight, dtype) = spacing between adjacent representable dtype values near base_weight remainder_offset = remainder_value × precision_step(base_weight, dtype)
DtypeSignExponent bitsStored fraction bitsInitial posture
FP161510Required on dreedle path
BF16187Required on scruffy path
FP321823Reference and test path
Precision caveatThe remainder must not be described as literally appending hidden IEEE mantissa bits if the effective weight is consumed in FP16 or BF16. The baseline interpretation is a signed adjustment among locally representable values. Sub-ULP precision would require a higher-precision effective computation and is outside v0.10.

thog2 · Remainder Architecture Enhancement7 · Storage
7.0 Storage and bit economics

No wasted bits in the final representation

PyTorch does not provide a generally useful native trainable 2-, 3- or 4-bit signed tensor. The logical representation and physical storage representation are therefore deliberately separate.

Prototype

torch.int8

One byte per remainder value. Wasteful, simple, debuggable. Valid range is restricted logically by remainder_bit_width.

Final target

Packed torch.uint8

Sub-byte signed values packed into ordinary byte tensors, with explicit unpack, update and repack functions or kernels.

7.1 Fixed-width storage impact

Illustrative calculation uses the quoted legacy-sheet baseline: persistent state 14.84% of the dense FP16 equivalent before adding the remainder.

Optimizer-state arithmeticA persistent int8 accumulator per materialised weight would itself cost another 8 bits per weight and could erase most of the saving. Byte-sized per-weight optimiser state is therefore acceptable only in a prototype, not in the final architecture.

thog2 · Remainder Architecture Enhancement8 · Optimisation
8.0 Remainder optimisation

The largest unresolved design question

Backpropagation is not the conceptual obstacle. The obstacle is converting a continuous rejected gradient into useful updates of a signed integer object without maintaining a dense floating-point shadow or full AdamW state.

input: G_remainder[position] goal: update remainder_value[position] while preserving low-bit persistent storage

8.1 Best current baseline

A custom remainder updater should borrow the useful ideas of adaptive optimisation without inheriting AdamW’s storage assumptions.

CandidatePersistent stateStrengthWeakness
Stochastic integer updateNone beyond remainderMemory-cleanNoisy
Packed fractional-step accumulatorSub-byte per weightPreserves small gradientsNew quantised-state design
Block-shared momentumSmall per blockCheap smoothingLoses per-weight adaptation
Full Adam-like momentsTwo dense statesFamiliar optimisationUnacceptable target
Recommended prototype posturePermit an intentionally wasteful accumulator or float shadow only long enough to establish whether the remainder learns. Treat it as scientific scaffolding, never as the target architecture.
Best guessThe final updater will likely combine packed remainder values, stochastic or fixed-point fractional accumulation, and at most block-level adaptive state. It should be a separate optimiser coordinated with AdamW, not an AdamW subclass.

thog2 · Remainder Architecture Enhancement9 · Requirements
9.0 Requirements specification

Functional, numeric and memory requirements

REQ-FUNC-001 · Mandatory

The implementation shall maintain exactly one effective weight value at each transformer weight position.

REQ-FUNC-002 · Mandatory

The effective weight shall equal the materialised base weight plus the decoded remainder offset.

REQ-FUNC-003 · Mandatory

The remainder object shall be indexed by canonical materialised weight position, not by spectral coefficient position or transient kernel layout.

REQ-FUNC-004 · Mandatory

remainder_bit_width = 0 shall disable the enhancement and reproduce the current THOG2 path.

REQ-FUNC-005 · Mandatory

The baseline implementation shall support logical widths 0, 1, 2, 3, 4, 6 and 8 bits. Eight-bit mode shall be classified as diagnostic.

REQ-NUM-001 · Mandatory

Remainder values shall be signed integers using a documented two’s-complement logical range.

REQ-NUM-002 · Mandatory

The remainder offset shall be a signed low-order precision offset whose unit is defined mechanically from the effective-weight dtype at the local base-weight scale.

REQ-NUM-003 · Mandatory

FP16 and BF16 precision-step semantics shall be implemented and tested independently. FP32 shall be available as a reference path.

REQ-NUM-004 · Prohibition

The baseline specification shall not claim that the remainder literally appends hidden mantissa bits while the resulting effective weight remains in the same lower-precision dtype.

REQ-OPT-001 · Mandatory

The existing base-coefficient optimiser path shall remain AdamW unless separately changed by another specification.

REQ-OPT-002 · Mandatory

The baseline gradient router shall give the base path the projection of the effective-weight gradient into the current basis span and give the remainder path the rejected component.

REQ-MEM-001 · Prohibition

The final architecture shall not retain a persistent dense FP16 or FP32 remainder shadow.

REQ-MEM-002 · Prohibition

The final architecture shall not retain full per-weight Adam first- and second-moment tensors for the remainder object.

REQ-STOR-001 · Mandatory

The prototype may use torch.int8 remainder values. The final representation shall support sub-byte packing in torch.uint8 storage without a new PyTorch dtype.

What is settled—and what is not

AreaStatusCurrent position
Two learned objects / one effective fieldSettledCore architecture.
Materialised-weight indexingSettledCanonical weight position is ground truth.
One backpropagationSettledSingle effective-weight gradient feeds both paths.
Gradient routingBaselineBase projection first; rejection to remainder.
FP16/BF16 numeric unitSpecify preciselyAdjacent representable-value spacing at local scale.
Final remainder updaterLargest unknownMust be adaptive enough to learn and lean enough to preserve compression.
Required bit widthEmpiricalOffline and training sweeps required.
Packed variable-width blocksLaterDo not obscure first fixed-width experiments.

Recommended sequence

  1. Freeze this architecture and requirements specification after review.
  2. Build an offline representation diagnostic using fixed remainder widths.
  3. Build a deliberately simple torch.int8 training prototype.
  4. Use a scientifically permissive updater to establish whether the remainder signal helps.
  5. Only after a positive result, design the packed and memory-clean updater.
  6. Add variable-width storage only after the fixed-width bit-quality curve is understood.
Brutal decision ruleDo not spend serious effort on packed optimiser plumbing unless 2–4 logical bits produce a convincing quality improvement.

Appendix A

Controlled terminology

base_coefficients
The compact spectral learned object.
base_weight
The transient weight value materialised from the base coefficients.
remainder_value
The signed low-bit logical integer associated with one materialised weight position.
remainder_offset
The decoded numeric signed low-order precision offset.
effective_weight
The single weight consumed by the transformer: base weight plus remainder offset.
materialised-weight-indexed
Associated with canonical materialised weight position rather than coefficient or kernel layout.
precision_step
The local least-significant-place unit implied by the effective-weight dtype and base-weight scale.
rejected gradient
The effective-weight gradient left after subtracting the projected component.

Conclusion

The Remainder Architecture Enhancement is coherent enough to move from conversation to controlled experimentation.

Its forward semantics, object association, one-backpropagation structure and baseline gradient routing are straightforward. Its scientific value depends on whether a small number of bits repairs model quality. Its engineering value depends on whether those bits can be trained without persistent optimiser state becoming larger than the remainder itself.

The architecture succeeds only if the remainder remains a small local correction—not a dense model or dense optimiser state in disguise.
