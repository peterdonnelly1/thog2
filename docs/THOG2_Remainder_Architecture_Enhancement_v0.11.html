1. Executive summary

THOG2 presently learns compact spectral coefficients and recreates transformer weights from them. The Remainder Architecture Enhancement adds a second learned object containing signed, low-order precision offsets at materialised weight positions.

The transformer still consumes exactly one value at every weight position: effective_weight = base_weight + remainder_offset.

The architecture therefore has two learned stored objects, base_coefficients and remainder_values, but only one effective weight field during execution.

Base object

The existing spectral learned object. It captures structured, basis-expressible weight content and continues to use the existing THOG2 optimiser path.

Remainder object

A signed low-bit object indexed by canonical materialised weight position. It supplies local low-order precision adjustments that the base geometry may not express independently.

Central hypothesisTHOG2 may already be learning a useful structured approximation to dense weights, but may be missing a small number of independent low-order bits at each materialised weight position.
Central riskThe remainder may require too many bits, or its optimiser may require persistent per-weight state large enough to erase the compression saving.

2. Concept

The remainder path is not a residual connection, not a second dense model, and not a replacement basis. It is a separate low-bit learned object whose sole role is to make local corrections to materialised base weights.

base_weight      = materialise(base_coefficients)
remainder_offset = decode(remainder_value, base_weight, effective_dtype)
effective_weight = base_weight + remainder_offset

2.1 Design intent

  • The base path carries structured, smooth, compressible weight content.
  • The remainder path carries local signed low-order precision corrections.
  • The transformer computation sees only the summed effective weight.
  • The remainder association is independent of Chebyshev, DCT, or any future basis family.

2.2 Name and controlled phrase

The enhancement should be referred to as thog2: “Remainder Architecture Enhancement”. The preferred technical phrase for the correction term is signed low-order precision offset.

3. Proposed architecture

3.1 Persistent learned state

base_coefficients
base_optimizer_state

remainder_values
remainder_metadata
optional_remainder_update_state

The following shall not be introduced in the final architecture:

persistent dense effective weights
persistent fp16/fp32 remainder shadow
dense AdamW first/second moment state for every materialised weight

3.2 Materialised-weight indexing

Remainder values are materialised-weight-indexed. They are associated with the canonical position of a materialised dense-equivalent weight, not with a Chebyshev coefficient and not with an implementation-specific materialisation order.

Tensor formExample canonical address
Ordinary matrix(layer, tensor_family, row, column)
Head-aware attention(layer, tensor_family, head, role, row, column)

3.3 Required invariant

one canonical materialised weight position
    -> one base value
    -> one remainder value or explicit zero
    -> one effective value
Addressing failure is catastrophicIf a remainder value intended for one q/k/v/head/layer coordinate is applied elsewhere, training can look superficially plausible while being functionally meaningless.

3.4 Forward pass

base_weight      = basis.materialise(base_coefficients)
remainder_value  = remainder_store[materialised_weight_position]
remainder_offset = remainder_value * precision_step
effective_weight = base_weight + remainder_offset

output = transformer_operation(input, effective_weight)

3.5 Loss and backpropagation

The loss remains the ordinary language-model loss. There is no second forward pass, no second loss, and no second backpropagation pass.

loss = language_model_loss(outputs, targets)

G_effective = d(loss) / d(effective_weight)

4. Numeric semantics

The remainder value is a signed low-bit integer. Its numeric meaning must be mechanically tied to the least significant representable place of the selected effective-weight dtype at the local scale of the base weight.

precision_step(base_weight, dtype)
    = spacing between adjacent representable dtype values near base_weight

remainder_offset

Precision caveatThe remainder should not be described as literally appending hidden IEEE mantissa bits if the final effective weight is consumed in the same lower-precision dtype. The baseline interpretation is a signed adjustment among locally representable values.

5. Storage and bit economics

PyTorch does not provide a generally useful native trainable 2-, 3-, or 4-bit signed tensor. The logical representation and the physical storage representation are therefore deliberately separate.

StagePhysical storageRationale
Prototypetorch.int8 remainder_valuesSimple, debuggable, wasteful. The logical range enforces 2/3/4-bit behaviour.
Final targettorch.uint8 packed_remainder_valuesMultiple signed low-bit values packed into byte storage. No custom PyTorch dtype required.

5.1 Fixed-width storage impact

Illustrative calculation using the quoted legacy-sheet baseline: persistent state 14.84% of the dense FP16 equivalent before adding the remainder.

Storage warningA byte-sized persistent update accumulator per materialised weight would itself cost another 8 bits per weight. That may be acceptable for a prototype, but it is not acceptable as the default final architecture.

6. Remainder optimisation

This is the largest unresolved design question. Backpropagation is not the hard part. The hard part is converting a continuous rejected gradient into useful low-bit integer updates without persistent dense float state.

6.1 Gradient routing

The base path should receive the component of the effective-weight gradient expressible by the current basis. The remainder path should receive the rejected component.

G_base      = project_to_base_span(G_effective)
G_remainder = G_effective - G_base

Informally: the base path gets first bite; the remainder receives the crumbs.

6.2 Baseline optimiser concept

RemainderOptimizer:
    persistent:
        remainder_values
        optional small update state

    input:
        G_remainder
        precision_step
        remainder_lr

    update:
        desired_value_change = -remainder_lr * G_remainder / precision_step
        convert desired movement to integer up / hold / down
        clamp to signed b-bit range
Candidate mechanismPersistent stateStrengthWeakness
Stochastic integer updateNone beyond remainderMemory-cleanNoisy
Packed fractional-step accumulatorSub-byte per weightPreserves small gradientsRequires new quantised-state design
Block-shared momentumSmall per blockCheap smoothingLoses per-weight adaptation
Full Adam-like momentsTwo dense statesFamiliar optimisationLikely unacceptable
Recommended postureDo not start by cloning PyTorch AdamW. Borrow the mathematical ideas only where needed. The remainder object has different assumptions: integer-valued parameter, continuous gradient, quantised update, tiny persistent state.

7. Requirements specification

7.1 Functional requirements

REQ-FUNC-001

The implementation shall maintain exactly one effective weight value at each transformer weight position.

REQ-FUNC-002

The effective weight shall equal the materialised base weight plus the decoded remainder offset.

REQ-FUNC-003

The remainder object shall be indexed by canonical materialised weight position, not by spectral coefficient position or transient kernel layout.

REQ-FUNC-004

remainder_bit_width = 0 shall disable the enhancement and reproduce the current THOG2 path.

REQ-FUNC-005

The baseline implementation shall support logical widths 0, 1, 2, 3, 4, 6 and 8 bits. Eight-bit mode shall be classified as diagnostic.

7.2 Numeric requirements

REQ-NUM-001

Remainder values shall be signed integers using a documented two’s-complement logical range.

REQ-NUM-002

The remainder offset shall be a signed low-order precision offset whose unit is defined mechanically from the effective-weight dtype at the local base-weight scale.

REQ-NUM-003

FP16 and BF16 precision-step semantics shall be implemented and tested independently. FP32 shall be available as a reference path.

7.3 Optimisation and memory requirements

REQ-OPT-001

The existing base-coefficient optimiser path shall remain AdamW unless separately changed by another specification.

REQ-OPT-002

The baseline gradient router shall give the base path the projection of the effective-weight gradient into the current basis span and give the remainder path the rejected component.

REQ-MEM-001

The final architecture shall not retain a persistent dense FP16 or FP32 remainder shadow.

REQ-MEM-002

The final architecture shall not retain full per-weight Adam first- and second-moment tensors for the remainder object.

REQ-STOR-001

The prototype may use torch.int8 remainder values. The final representation shall support sub-byte packing in torch.uint8 storage without a new PyTorch dtype.

9.1 Recommended development sequence

  1. Freeze this architecture and requirements specification after review.
  2. Build an offline representation diagnostic using fixed remainder widths.
  3. Build a deliberately simple torch.int8 training prototype.
  4. Use a scientifically permissive updater to establish whether the remainder signal helps.
  5. Only after a positive result, design the packed and memory-clean updater.
  6. Add variable-width storage only after the fixed-width bit-quality curve is understood.
Brutal decision ruleDo not spend serious effort on packed optimiser plumbing unless 2–4 logical bits produce a convincing quality improvement.

10. Controlled terminology

base_coefficientsThe compact spectral learned object.
base_weightThe transient weight value materialised from the base coefficients.
remainder_valueThe signed low-bit logical integer associated with one materialised weight position.
remainder_offsetThe decoded numeric signed low-order precision offset.
effective_weightThe single weight consumed by the transformer: base weight plus remainder offset.
materialised-weight-indexedAssociated with canonical materialised weight position rather than coefficient or kernel layout.
precision_stepThe local least-significant-place unit implied by the effective-weight dtype and base-weight scale.
projected gradientThe portion of the effective-weight gradient expressible by the current base geometry.
rejected gradientThe effective-weight gradient left after subtracting the projected component.

Conclusion

The Remainder Architecture Enhancement is coherent enough to move from discussion to controlled experimentation. Its forward semantics, object association, one-backpropagation structure and baseline gradient routing are straightforward. Its scientific value depends on whether a small number of bits repairs model quality. Its engineering value depends on whether those bits can be trained without persistent optimiser state becoming larger than the remainder itself.

The architecture succeeds only if the remainder remains a small local correction, not a dense model or dense optimiser state in disguise.
