1. Executive summary

THOG2 currently learns compact spectral coefficients and recreates transformer weight matrices on demand. The enhancement adds one signed low-bit value at each canonical materialised weight position. The transformer continues to consume one effective weight field.

effective_weight = base_weight + remainder_offset

The base object carries structured, basis-expressible content. The remainder object carries only local low-order corrections that the current base geometry cannot independently express.

Non-negotiable economic constraint. A 4-bit remainder plus byte-sized per-weight optimiser state is not meaningfully a 4-bit architecture. The final optimiser may not retain a dense float shadow or default byte-per-weight state.

2.1 Covered matrices

The initial implementation covers spectrally generated q, k, v, attention-output, MLP-expansion and MLP-contraction matrices. Biases, embeddings, layer-normalisation parameters, unrelated LM-head parameters and dense-model weights are excluded.

2.2 Geometry contract

All registered spectral geometries must expose the same final two-dimensional matrix and the projection interface in Section 6. No geometry-specific remainder storage layout is permitted.

The association must not depend on qkv packing, materialisation order, vectorisation, kernel layout, head reshaping or checkpoint segmentation.

4. Numeric encoding and decoding

4.1 Signed range

q_min = -(2 ** (bit_width - 1))
q_max =  (2 ** (bit_width - 1)) - 1

Width zero disables the architecture. Required active widths are 1, 2, 3, 4, 6 and 8.

4.2 Fixed v0.12 decoder

v0.12 uses adjacent_representable. Positive codes step upward through adjacent values representable in the effective dtype; negative codes step downward.

decode_adjacent(base, q, dtype):
    x = cast(base, dtype)
    if q > 0:
        repeat q times: x = nextafter(x, +infinity)
    if q < 0:
        repeat abs(q) times: x = nextafter(x, -infinity)
    return x

This literal definition governs zero, subnormals and exponent boundaries. Any accelerated decoder must be bitwise identical.

4.3 Straight-through base derivative

decoded = decode_adjacent(base_weight, remainder_value, effective_dtype)

effective_weight =
    base_weight
    + stop_gradient(cast(decoded, base_weight.dtype) - base_weight)

The forward value is decoded. Before routing, the derivative with respect to the base weight is the identity.

4.4 Directional code step

step_up   = decode(base, q + 1) - decode(base, q)
step_down = decode(base, q) - decode(base, q - 1)

directional_step =
    step_up   for intended positive code movement
    step_down for intended negative code movement

Outward movement at either code bound is masked.

Precision meaning. v0.12 selects nearby values representable in the existing compute dtype. It does not claim to append hidden sub-ULP mantissa bits. Sub-ULP carrier precision is deferred.

5. Forward path

base_weight = materialiser.materialise(tensor_key, base_coefficients)
q = remainder_store.get(tensor_key, shape=base_weight.shape)
effective_weight = build_effective_weight(base_weight, q, effective_dtype)
output = linear(input, effective_weight, bias)

5.1 Dtype order

  1. Materialise through the existing THOG2 path.
  2. Cast a detached base copy to the resolved effective dtype.
  3. Decode the remainder in that dtype.
  4. Cast the decoded value back to the materialised carrier for the straight-through expression.
  5. Use the existing linear/autocast path.

5.2 Disabled path

if remainder_bit_width == 0:
    effective_weight = base_weight
    allocate no remainder state
    install no remainder hooks

6. Backward path and projection contract

G_effective = d(loss) / d(effective_weight)

G_coefficients, G_projected = projector.project_weight_gradient(
    tensor_key,
    G_effective,
    projection_context,
)

G_remainder = G_effective - G_projected
remainder_optimizer.observe(base_weight, q, G_remainder, ...)
return G_projected

The effective-weight hook consumes the full weight gradient, records or applies the remainder signal, and returns only the projectable component to existing autograd.

6.1 Required projector interface

project_weight_gradient(
    tensor_key,
    weight_gradient,
    projection_context,
) -> (coefficient_gradient, projected_weight_gradient)

coefficient_gradient is the exact reverse contraction or analysis transform for the materialiser. projected_weight_gradient is obtained by rematerialising that coefficient gradient.

6.2 Orthonormal CHEBY/DCT case

coefficient_gradient = B_transpose(weight_gradient)
projected_gradient   = B(coefficient_gradient)

Tensor-product geometries contract exactly the axes compressed in the forward path and preserve uncompressed axes. A future non-orthonormal basis must register a true least-squares projector; it may not silently use B B^T.

6.3 Gradient ownership

With the hook backend, autograd receives G_projected and generates coefficient gradients. The explicitly calculated coefficient gradient is for validation and the later streamed backend; it must not be applied twice.

7. R1 reference optimiser

R1 deliberately ignores optimiser memory so that optimiser inadequacy cannot be confused with representational failure.

7.1 Persistent state

remainder_values: int8
remainder_master: float32, same shape
AdamW first moment: float32, same shape
AdamW second moment: float32, same shape

R1 requires gradient accumulation equal to 1 and must report its true byte cost.

7.2 Master gradient

intended_direction = -sign(G_remainder)
directional_step = resolve_directional_step(base_weight, q, intended_direction)
G_master = G_remainder * directional_step
G_master = mask_outward_gradient_at_bounds(G_master)

7.3 AdamW and quantisation

AdamW(
    remainder_master,
    lr=1e-3,
    betas=(0.9, 0.999),
    eps=1e-8,
    weight_decay=0.0,
)

remainder_master = clamp(remainder_master, q_min, q_max)
remainder_values = clamp(round_to_nearest_even(remainder_master), q_min, q_max).int8()
Reference-only. R1 is an optimisation oracle and debugging aid. It is not the deployable architecture.

8. R2 target remainder optimiser

R2 is a separate optimiser coordinated with base AdamW. It uses factored second moments and stochastic integer updates, with no dense per-weight float shadow or moments.

8.1 Persistent state per matrix

remainder_values: low-bit logical values
row_second_moment: float32[out_features]
column_second_moment: float32[in_features]
optimizer_step: int64
RNG state

8.2 Required defaults

remainder_lr = 0.05
remainder_beta2 = 0.999
remainder_eps = 1e-8
remainder_clip_threshold = 1.0
remainder_max_code_step = 1
remainder_weight_decay = 0.0
remainder_rounding = stochastic

8.3 Code-space gradient

intended_direction = -sign(G_remainder)
directional_step = resolve_directional_step(base_weight, q, intended_direction)
G_code = G_remainder.float32 * directional_step.float32
G_code = mask_outward_gradient_at_bounds(G_code)

8.4 Factored second moment

row_obs = mean(G_code ** 2, dim=1)
col_obs = mean(G_code ** 2, dim=0)

row_v = beta2 * row_v + (1 - beta2) * row_obs
col_v = beta2 * col_v + (1 - beta2) * col_obs

normaliser = outer(row_v, col_v) / max(mean(row_v), eps)
U = G_code / sqrt(normaliser + eps)

u_rms = sqrt(mean(U ** 2))
U = U / max(1.0, u_rms / clip_threshold)

8.5 Integer update

delta_code = clamp(-remainder_lr * U, -1, +1)
probability = abs(delta_code)
integer_step = sign(delta_code) * (rand() < probability).int8()
remainder_values = clamp(remainder_values + integer_step, q_min, q_max)

Each covered weight moves down one code, holds, or moves up one code per completed update. v0.12 deliberately omits a dense first moment. If smoothing is necessary, the next permitted experiment is block-shared or low-rank momentum.

Rationale. This retains Adam-like adaptive scaling through factored second moments, represents tiny desired updates statistically rather than with a dense debt tensor, and keeps persistent optimiser state at O(rows + columns).

9. Gradient accumulation and non-finite handling

9.1 R2 microsteps

R2 must not retain a dense float remainder gradient across microsteps. Remainder values remain unchanged during the accumulation window.

micro_delta = full_delta_code / gradient_accumulation_steps
micro_sample = stochastic_round(micro_delta)

pending_integer_step = clamp(
    pending_integer_step + micro_sample,
    -remainder_max_code_step,
    +remainder_max_code_step,
)

pending_integer_step is transient int8 or packed low-bit state and is cleared after the completed optimiser update. Row/column observations are averaged across microsteps and persistent factors are updated once.

9.2 Atomic update order

1. finish all microstep backward passes
2. verify the whole update is finite
3. base AdamW step
4. apply pending remainder integer steps
5. update persistent remainder factors
6. clear transient state and gradients
7. increment completed_updates

9.3 Non-finite skip

If loss, required base gradients, remainder observations or factor updates are non-finite, both optimiser paths are skipped atomically. Pending steps and factor observations are discarded. The pre-update remainder RNG state is restored so uninterrupted and resumed runs remain identical.

10. Autograd and memory backends

10.1 Mandatory first backend

The first implementation uses effective_weight.register_hook(...). PyTorch may transiently materialise the full matrix gradient. The hook consumes it immediately, returns the projected component, and retains no dense float remainder gradient after returning.

10.2 Streamed backend

If measured peak memory is unacceptable, a custom autograd backend performs two tiled passes:

Pass A:
    recompute G_effective tiles
    accumulate coefficient gradient
    accumulate row/column squared-gradient observations

Pass B:
    recompute G_effective tiles
    materialise G_projected tiles
    G_remainder_tile = G_effective_tile - G_projected_tile
    generate pending integer updates
    discard each tile

Return grad_input and publish coefficient gradient.

The streamed backend is a major work item and should follow scientific proof unless the hook backend is unusable even for the proof.

12. Normative requirements

REQ-ARCH-001The transformer shall consume exactly one effective value at every covered position.
REQ-ARCH-002The remainder shall be indexed by canonical final two-dimensional matrix coordinate.
REQ-ARCH-003The active architecture shall use one forward pass, one loss and one backward pass.
REQ-NUM-001The required decoder shall be repeated adjacent-representable stepping in the selected dtype.
REQ-NUM-002Any accelerated decoder shall be bitwise identical to the literal reference decoder.
REQ-GRAD-001The base path shall receive the exact basis-projectable gradient component.
REQ-GRAD-002The remainder path shall receive the effective gradient minus that component.
REQ-OPT-001R1 shall implement the reference AdamW contract in Section 7.
REQ-OPT-002R2 shall implement the factored-RMS stochastic contract in Section 8.
REQ-OPT-003R2 shall retain no per-weight floating master, first moment or second moment.
REQ-MEM-001R2 persistent optimiser state shall scale as O(rows + columns) per matrix.
REQ-MEM-002Dense effective weights and gradients may be transient but shall not be checkpointed or retained.
REQ-COMPAT-001Width zero shall preserve the existing THOG2 path without remainder allocation or hooks.
REQ-COMPAT-002Non-finite updates shall skip base and remainder atomically.

13. Tests and acceptance criteria

13.1 Decoder and address tests

  • Code zero is bitwise identical to the base cast to the effective dtype.
  • Every signed code matches literal repeated nextafter.
  • Tests cover ±0, subnormals, exponent boundaries and maximum finite values.
  • Changing one code changes exactly one intended matrix coordinate.
  • All six tensor families and first/middle/final layers are covered.
  • Checkpoint round-trip preserves all codes and keys exactly.

13.2 Projection tolerances

relative_reconstruction_error <= 1e-6  # FP32
orthogonality_error          <= 1e-5  # FP32
integration tolerance        <= 1e-3  # FP16/BF16

13.3 Optimiser tests

  • R1 decreases a deterministic quadratic objective.
  • R2 never updates outward at code bounds.
  • R2 is reproducible from saved RNG state.
  • Mean stochastic movement matches requested movement within 3% relative or 0.01 absolute code step over a fixed seed ensemble.
  • Skipped non-finite updates leave parameters, codes, factor state and RNG unchanged.
  • Fresh → save → resume matches uninterrupted execution.

13.4 Memory gate

target persistent bits per covered weight
    <= remainder_bit_width + 0.5

The int8 prototype is exempt but must report theoretical packed storage separately.

13.5 Scientific gate

A 4-bit R1 remainder must, on a matched-token controlled sweep, either close at least 25% of the validation-loss gap to the selected dense baseline or improve validation loss by at least 0.05 over width zero. The result must hold for at least two of three seeds without sustained saturation above 10%.

Stop rule. If 4-bit R1 fails, do not implement packed storage or streamed autograd. If R1 passes but R2 fails, the optimiser becomes the research target.

14. Implementation sequence

Stage 0 — Contracts and offline diagnostic

Implement keys, canonical matrix views, literal decoder, projector interface and fixed-width diagnostic. Exit when decoder and projection tests pass.

Stage 1 — Forward path

Add int8 buffers and effective-weight construction for all six matrix families. Exit when zero-code equivalence and existing regressions pass.

Stage 2 — R1 reference training

Add full-matrix hook, gradient routing, FP32 master, AdamW, checkpointing and diagnostics. Exit when R1 learns on toy and short LM runs.

Stage 3 — R2 target optimiser

Add factored moments, stochastic updates, accumulation pending state, RNG checkpointing and atomic skip. Exit when optimiser, resume and memory-state tests pass.

Stage 4 — Controlled sweep

Compare widths 0,1,2,3,4,6,8 and compare R1/R2 at widths 2 and 4. Apply the scientific gate.

Stage 5 — Packed storage and streamed backward

Implement only after Stage 4 demonstrates clear value or the hook backend fails the memory gate.
