1 Purpose and operating principle

1.1 Purpose

This implementation plan is the companion to the THOG2 Chebyshev Sheet Requirements Specification v0.25. Its job is not to restate the architecture. Its job is to turn that architecture into a low-risk sequence of code changes that preserves every currently working non-dense geometry while adding the new semantic sheet geometries.

The plan assumes the existing dense path remains a control path, not part of the compact geometry system. The compact system must remain coherent: depth, generic sheet-column, generic sheet-row, MLP block, head-aware block, full block, and semantic sheets must all resolve through one component catalogue.

1.2 Implementation principle

The implementation shall be staged so that each stage has a small surface area, a clear regression gate, and a visible reason to exist. The most important sequencing decision is to refactor the resolver and trajectory spine before adding new geometry. Adding semantic sheet cases into the current two-slot resolver would reproduce the problem this work is meant to fix.

Blunt rule: do not add sheet_mlp, sheet_qkv, or sheet_out_* directly into the old attention_geometry + mlp_geometry split. That would be expedient but structurally wrong.

1.3 What this plan deliberately avoids

This plan does not chase direct/factorised hot paths first. Correct materialisation and gradient flow come first. Direct execution paths may be added only after materialise-then-linear equivalence is tested. This plan also does not redesign embeddings, positional embeddings, attention kernels, layernorm vectors, optimiser policy, resume semantics, or checkpoint format beyond the identity fields needed for compact geometry.

2 Current codebase assessment

The codebase is ready for the additions, but only after a small integration pass. The useful thing is that repeated-weight logic is already isolated into trajectory classes. The weak thing is that the resolver is still organised around only two component slots.

2.1 Useful existing structure

The current trajectory classes already expose the right broad interface: materialisation by family name, vector materialisation, semantic parameter enumeration, and parameter reports. The implementation should preserve that external shape. The model should not need to learn the internal coefficient layout of each geometry.

2.2 Main abstraction gap

The abstraction gap is the resolver. The new architecture has three independent compact component slots: MLP, QKV, and attention-output. The current resolver has attention and MLP only. The implementation therefore starts by making the resolver more precise, then makes the model consume a trajectory factory rather than choosing trajectory classes directly.

3 Target implementation shape

3.1 Public interface

The public interface should remain simple. Users should normally select one --geometry-preset. Existing lower-level --attention-geometry and --mlp-geometry options may remain for compatibility, but new semantic sheet work should be driven by named presets rather than by asking users to compose internals manually.

3.1.1 Existing presets legacy_sheet_col, depth, mlp_block, head_aware_block, and full_block shall remain accepted.

3.1.2 generic_sheet_col shall become the canonical new name for legacy_sheet_col.

3.1.3 New public presets shall not require users to specify separate internal component flags.

3.3 Trajectory/provider interface

The cleanest implementation is a composed trajectory object that presents the same public interface as the existing trajectories while dispatching each repeated matrix family to the provider selected by the resolver. Existing trajectory classes can be wrapped or delegated to first; they do not need to be rewritten up front.

4 Optimal staging

The plan uses six stages. Fewer stages would hide too many failure modes; more stages would create ceremony. The ordering is deliberate: resolver first, old-path composition second, simplest new geometry third, MLP fourth, attention fifth, public integration last.

4.1 Stage 1 - resolver, names, aliases, identity

4.1.1 Add canonical constants for generic_sheet_col while keeping legacy_sheet_col as an accepted input alias.

4.1.2 Introduce ResolvedCompactGeometry or equivalent. It must include mlp_component, qkv_component, and attn_out_component.

4.1.3 Keep the old ResolvedCompactSelectors available temporarily or provide a compatibility shim so existing callers do not break in the same commit.

4.1.4 Update compact_identity_metadata, artifact identity fields, and run_config canonical dictionaries to store requested and canonical names.

4.1.5 No trajectory code should change in this stage.

4.2 Stage 2 - composed trajectory spine with old geometries only

4.2.1 Create a trajectory factory, for example build_compact_trajectory(config, resolved_geometry, ...).

4.2.2 Create a ComposedTrajectory facade or equivalent provider map that supports materialize(), materialize_vector(), named_semantic_parameters(), family_report(), and count methods.

4.2.3 Initially route existing presets to the existing concrete trajectory classes with no new geometry.

4.2.4 Replace SheetGPT constructor branching with the factory. This is the main integration cut.

4.2.5 Gate this stage by proving old presets are equivalent before and after the factory cutover.

4.3 Stage 3 - generic_sheet_row

4.3.1 Implement generic_sheet_row as the transposed generic ablation for repeated matrix families: coeff[in, P, q_out].

4.3.2 Use an output-row basis with width equal to output_rows. Do not compress vectors through this path unless explicitly specified later.

4.3.3 Expose generic_sheet_row as a public preset only after materialisation, direct_value, parameter count, and gradient-flow tests pass.

4.3.4 This stage exercises the new provider spine with the least semantic complexity.

4.4 Stage 4 - sheet_mlp

4.4.1 Implement sheet_mlp up and down as semantic sheets with hidden_channel whole and d_model compressed.

4.4.2 For W_up use coeff[hidden, P, X]. For W_down use coeff[hidden, P, X] over d_model_output.

4.4.3 Do not reuse the current direct_factorised_mlp path unless the provider explicitly advertises support. Start with materialize-then-linear.

4.4.4 Expose sheet_mlp with generic_sheet_col attention fallback.

4.5 Stage 5 - semantic attention sheets

4.5.1 Implement sheet_qkv so that qkv_role, head, and local_head_channel remain whole; only depth and d_model_input are compressed.

4.5.2 The provider must materialize the packed legacy attention_input_weight expected by the current attention hot path.

4.5.3 Implement sheet_out_dmodel and sheet_out_local as separate providers or provider modes. Do not collapse them prematurely.

4.5.4 Expose sheet_qkv, sheet_attn_dmodel, and sheet_attn_local once role/head/local mapping tests pass.

4.6 Stage 6 - preset integration, accounting, wrappers, smoke gates

4.6.1 Expose sheet_full_dmodel and sheet_full_local, plus diagnostic mixed presets only after component-level tests pass.

4.6.2 Update wrapper help text and preset validation lists. Do not change existing run strings unnecessarily.

4.6.3 Extend parameter_report and family_report to include component name, coefficient shape, dense-equivalent shape, whole axes, compressed axes, and consumed orders.

4.6.4 Run the full unit suite and a compact smoke matrix. Only then start wall-clock or serious OWT comparisons.

5.2 Smoke-run matrix

The smoke matrix should use tiny dimensions and one or two updates. It is not a quality benchmark; it is a wiring and gradient-flow gate.

python -m unittest discover tests

for preset in generic_sheet_col depth mlp_block head_aware_block full_block generic_sheet_row sheet_mlp sheet_qkv sheet_attn_dmodel sheet_attn_local sheet_full_dmodel sheet_full_local; do
  python -m run_thog2_owt \
    --model-type sheet --run-mode fresh --host-label smoke \
    --run-name IMPL_SMOKE_${preset} --dataset openwebtext --data-dir data/openwebtext \
    --max-iters 2 --eval-interval 1 --eval-iters 1 \
    --batch-size 2 --gradient-accumulation-steps 1 --block-size 16 \
    --n-layer 2 --n-head 2 --n-embd 64 \
    --o-depth 2 --o-attn-d-model 8 --o-attn-qkv-per-channel 2 \
    --o-attn-out-per-channel 2 --o-mlp-d-model 8 --o-mlp-hidden 16 \
    --geometry-preset ${preset} --basis-family chebyshev --wandb-mode disabled \
    --device cuda --dtype bfloat16
done

5.3 Benchmark gate

Only after the smoke matrix passes should serious OWT comparisons run. The first serious ablation set should be generic_sheet_col, generic_sheet_row, depth, sheet_mlp, sheet_qkv, sheet_attn_dmodel, sheet_attn_local, sheet_full_dmodel, and sheet_full_local. Equal-wall comparison remains the priority metric, with validation loss as the main result and throughput/memory as diagnostics.

6.1.1 All nanoGPT/THOG code edits shall preserve existing working code paths unless a stage explicitly replaces a path after tests pass.

6.1.2 New code in nanoGPT-derived files shall continue to use THOG marking comments according to the project convention.

6.1.3 Do not delete existing nanoGPT lines as part of this work. Comment out replaced lines and mark the new block.

6.1.4 No semantic sheet preset shall introduce basis sharing across qkv_role or head.

6.1.5 No semantic MLP sheet preset shall introduce basis sharing across hidden_channel.

7 Definition of done

The implementation is done when all six stages have landed, all existing compact presets still run, all new serious presets run, parameter reports are trustworthy, checkpoint identity is unambiguous, and the smoke matrix passes without non-finite recovery or shape exceptions.

The implementation is not done merely because the model starts. It must produce correct materialised weights, route gradients to the right coefficients, preserve aliases, expose shape/accounting metadata, and support a clean equal-wall benchmark grid.
