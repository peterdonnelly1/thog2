1 Concept and architecture

1.1 Purpose and authority

This document is the requirements source for the next THOG2 semantic-sheet geometry pass. It does not replace the dense GPT control path. It governs the non-dense compact geometries that regenerate repeated transformer-block weights from learned compact coefficient tensors and deterministic orthogonal bases.

The immediate implementation goal is not to invent a new training loop or a new language-modeling task. The goal is to make the compact geometry system coherent enough that depth, generic sheet, block, and semantic sheet variants are first-class alternatives in the same catalogue. The implementation must stop accreting one-off geometry classes whose differences are only discoverable by reading old files and wrapper history.

1.1.1 All non-dense geometry names shall resolve through one documented catalogue.

1.1.2 Dense shall remain a conventional control path and shall not masquerade as a compact geometry.

1.1.3 Existing working compact paths shall be preserved unless a branch is explicitly declared checkpoint-breaking.

1.2 Core compact-depth idea

THOG2 treats repeated transformer matrix families across layer depth as structured learned objects. Instead of storing an unrelated dense matrix for every layer, the model stores learned coefficients in a compact basis representation. At runtime, layer-specific dense-equivalent weights are reconstructed or applied directly from those coefficients.

Depth is privileged. Every compact geometry in this document has a depth basis. A depth-only geometry compresses only across layer depth. A sheet geometry compresses depth plus one matrix axis. A block geometry compresses depth plus both matrix axes. A semantic sheet geometry chooses that one matrix axis according to transformer semantics rather than blind matrix orientation.

1.3 Transformer interface and residual stream

The interface between token space and compact geometry remains the ordinary GPT residual stream. Token embedding and positional embedding produce dense residual-stream vectors. The compact representation is not a replacement tokenizer, not a separate token-space encoding, and not a Chebyshev representation of the input sequence. The compact basis acts on repeated parameter families, not on the raw token sequence.

1.4 Clean-axis rule

The central design rule is that a compact geometry must say which axes are compressed and which axes are left whole. Generic geometries may flatten a matrix into output rows and input columns. Semantic geometries must respect transformer structure: QKV role, attention head, local head channel, MLP hidden channel, and residual d_model channel are not the same thing.

1.5 Integrated non-dense geometry catalogue

The resolver shall decompose compact presets into three component selectors: mlp_component, qkv_component, and attn_out_component. Existing presets then become rows in the same catalogue as the new semantic sheet presets, not special historical cases.

3 Existing geometry ladder

Depth, generic sheet-column, MLP block, head-aware attention block, and full block remain first-class geometries. They are not to be treated as obsolete historical debris. They shall route through the same component resolver used by the semantic sheet additions.

Depth leaves both matrix axes whole and compresses only layer depth. Generic sheet-column leaves output rows whole and compresses depth plus input columns. MLP block and head-aware attention block compress both matrix axes in addition to depth. full_block composes the MLP block and head-aware attention block paths.

4 New semantic sheet geometry

Semantic sheet geometries keep declared semantic axes whole and compress depth plus exactly one selected matrix axis. The point is to isolate whether semantically meaningful sheet structure can preserve useful representation while reducing persistent parameter memory.

4.1.1 sheet_mlp up shall represent W_up[hidden, d_model_input] with coeff[hidden, P, X].

4.1.2 sheet_mlp down shall represent W_down[d_model_output, hidden] with coeff[hidden, P, X].

4.2.1 sheet_qkv shall represent W_qkv[qkv_role, head, local_head_channel, d_model_input] with coeff[qkv_role, head, local_head_channel, P, Q].

4.3.1 sheet_out_dmodel shall represent W_out[d_model_output, head, local_head_channel] with coeff[head, local_head_channel, P, Q].

4.3.2 sheet_out_local shall represent W_out[d_model_output, head, local_head_channel] with coeff[d_model_output, head, P, O].

7.2.1 New runs shall store both requested_preset and canonical_preset.

7.2.2 legacy_sheet_col shall remain accepted indefinitely as an input preset and shall canonicalise to generic_sheet_col.

7.2.3 curve shall remain accepted as an input alias for depth where that alias is present in current user workflows.

8 Correctness, accounting, and non-goals

8.1.1 For every compact component, direct/factorised execution and materialise-then-linear execution, where both exist, shall produce equivalent outputs within dtype-appropriate tolerance.

8.1.2 Forward and backward paths shall send gradients to the intended coefficient tensors and not to stale dense fallback parameters.

8.3.1 No semantic sheet preset shall compress across qkv_role.

8.3.2 No semantic sheet preset shall compress across attention head.

8.3.3 No semantic MLP sheet preset shall compress across hidden_channel.

This specification does not require changing the tokenizer, embedding interface, objective, dataset, optimizer, scheduler, or evaluation protocol. It does require that compact metadata be explicit enough to reconstruct the experiment without reading wrapper archaeology.

9 Implementation-plan implications

The implementation plan should begin with resolver and identity hygiene, not with new materialisers. First canonicalise generic_sheet_col, preserve legacy_sheet_col, introduce the three-slot component resolver, and route existing presets through it unchanged. Only then add generic_sheet_row and the semantic sheet components.

The blunt criterion is this: after implementation, a reader should be able to look at a run artifact and know exactly which axes were compressed, which axes were whole, which compact orders were consumed, and which coefficient tensor shapes existed. If that is not true, the implementation has failed the integration goal even if the code runs.
